# FrameGrid 初步需求规划

## 1. 文档信息

- 项目名称：FrameGrid
- 文档性质：初步产品需求与技术规划
- 当前阶段：需求收敛与架构设计
- 产品形态：本地优先的像素角色动画生产工具

本文档用于确定 FrameGrid 的产品目标、核心流程、功能边界、数据结构、技术栈和阶段性开发计划。具体算法、界面原型和 API 设计将在后续技术设计文档中展开。

## 2. 产品定位

FrameGrid 是一套面向 2D 游戏角色的 AI 像素动画生产流水线。

它不是单纯的图片像素化工具，而是一个完整的角色资产生产系统，负责：

1. 创建和管理角色项目。
2. 根据文字描述生成角色标准参考图。
3. 管理同一角色的服装、装备和外观变体。
4. 根据动作描述生成关键动作帧。
5. 将模型生成图规格化为严格的像素资产。
6. 检测角色身份、尺寸、锚点和动作连续性。
7. 对失败帧进行人工修正或局部重新生成。
8. 组合动画并导出游戏引擎可直接使用的资源。

FrameGrid 的核心价值是让 AI 生成结果变得可约束、可检查、可修改、可复用，而不是期望模型一次生成完美动画。

## 3. 产品目标

### 3.1 核心目标

- 用户可以从一段角色描述开始创建完整角色项目。
- 同一角色的不同方向、动作和服装保持身份一致。
- 输出图片严格遵循统一的像素网格、画布、调色板和锚点。
- 每一个动作帧都可以独立替换，不需要重新生成全部内容。
- 生产过程中的原图、中间结果和最终结果均可追踪。
- 生图服务可以替换，不与某一家模型供应商绑定。
- 最终资产能够直接用于 Godot 等游戏引擎。

### 3.2 非目标

第一阶段不追求：

- 完全无人参与的一键式动画生成。
- 自动修复所有肢体结构、遮挡和身份变化问题。
- 通用骨骼动画编辑器。
- 复杂的像素画手绘软件能力。
- 3D 角色、3D 骨骼或实时渲染。
- 同时支持大量不同美术风格。

## 4. 核心概念

### 4.1 角色项目

角色项目是 FrameGrid 的最高层资源单位，包含：

- 角色描述。
- 像素规格。
- 角色标准参考图。
- 固定身体比例。
- 调色板。
- 锚点规则。
- 服装和装备变体。
- 动作集合。
- 生成历史和质检记录。
- 导出配置。

### 4.2 角色母版

角色母版是后续所有生成任务必须参考的标准资料，也可称为 Character Bible。

至少包含：

- 正面标准图。
- 背面标准图。
- 左侧标准图。
- 右侧标准图，或明确允许从左侧镜像。
- 游戏视角要求的俯视图或三分之四视图。
- 固定的头身比例、身体高度和躯干宽度。
- 发型、面部、服装和装备的文字描述。
- 不允许发生变化的身份特征。
- 主色、阴影色、描边色和透明色规则。
- 脚底、身体中心、手部和武器握点等锚点。

是否生成真正的头顶图应由项目视角决定。普通俯视角游戏通常更需要朝上、朝下和侧面，而不是真正垂直向下的头顶视图。

### 4.3 角色变体

变体用于描述同一角色的不同外观，例如：

- 默认服装。
- 冬季服装。
- 护甲。
- 不同武器。
- 受伤状态。

变体继承角色的基础身份、身体比例和锚点规则，但可以拥有独立参考图和调色板补充项。

第一阶段不要求将衣服拆成可实时组合的独立图层。优先将每个变体视为一套完整角色外观，以降低遮挡和边缘处理复杂度。

### 4.4 动作

动作由以下信息定义：

- 动作名称，例如 `idle`、`run`、`attack`、`dig`。
- 方向，例如 `down`、`up`、`left`、`right`。
- 总帧数。
- 播放速度。
- 是否循环。
- 关键姿势列表。
- 动作阶段，例如准备、蓄力、接触、回收。
- 允许的身体位移。
- 固定锚点和动态锚点。
- 武器和特效是否独立处理。

### 4.5 帧

每个帧必须同时保存：

- 原始生成图。
- 裁切或分割结果。
- 规格化结果。
- 人工修正结果。
- 当前采用的最终版本。
- 生成提示词和 Provider 信息。
- 参考图和前序帧来源。
- 锚点、包围盒、调色板和质检数据。
- 版本和替换历史。

## 5. 完整用户流程

### 5.1 创建角色项目

用户输入：

- 项目名称。
- 角色名称。
- 角色文字描述。
- 游戏视角。
- 目标帧画布。
- 目标角色占用尺寸。
- 逻辑像素尺寸。
- 美术风格。
- 所需方向。

系统创建一个可移植的本地项目目录，并生成初始配置。

### 5.2 定义像素规格

“32 像素”或“64 像素”必须拆分为以下参数：

- 帧画布尺寸，例如 `32x32`、`64x64`、`96x80`。
- 角色目标高度，例如在 `64x64` 画布中占用 48 像素。
- 逻辑像素倍率，例如每个逻辑像素输出为 `2x2` 实际像素。
- 目标调色板颜色数量。
- 脚底锚点位置。
- 透明边缘和 Alpha 规则。

系统提供预设，同时允许高级用户自定义。

### 5.3 生成角色母版

系统根据角色描述生成第一轮标准参考图。

建议生成顺序：

1. 先生成主视角。
2. 用户确认角色身份和美术方向。
3. 以确认后的主视角为条件生成其他方向。
4. 用户选择或重新生成不合格方向。
5. 锁定角色母版。

角色母版锁定后，后续动作生成默认不得修改角色身份特征。

### 5.4 创建外观变体

用户选择现有母版并描述变更内容，例如“更换为矿工服并增加头灯”。

系统生成变体参考图，并检查：

- 身体比例是否保持。
- 面部和发型是否保持。
- 新服装在各方向是否一致。
- 装备位置和尺寸是否一致。

### 5.5 创建动作

用户选择：

- 角色变体。
- 动作模板或自定义动作。
- 方向。
- 总帧数。
- FPS。
- 是否循环。
- 武器、道具和特效。

系统根据动作描述产生结构化动作计划，包括关键姿势、时间位置和动作约束。

### 5.6 生成关键动作帧

系统将以下内容交给 Image Provider：

- 角色母版。
- 当前变体参考图。
- 目标方向。
- 目标姿势描述。
- 前一关键帧。
- 动作阶段。
- 固定比例和禁止变化项。

关键帧应优先生成动作语义明确的姿势，不直接要求模型一次生成完整 Sprite Sheet。

### 5.7 生成中间帧

关键帧通过确认后，再生成中间帧。

中间帧生成需要参考：

- 前后关键帧。
- 角色母版。
- 插值位置。
- 锚点和位移规则。
- 动作循环要求。

### 5.8 自动规格化

脚本流水线依次执行：

1. 背景和主体分割。
2. Alpha 清理与二值化。
3. 帧裁切，但保留原始空间关系。
4. 映射到统一逻辑像素网格。
5. 使用最近邻方式还原目标尺寸。
6. 统一角色或变体调色板。
7. 按脚底或指定锚点对齐。
8. 规范画布尺寸。
9. 记录包围盒、重心和占用面积。

规格化只能处理机械规则，不能承诺自动修复身份变化、肢体错误、武器变形和遮挡错误。

### 5.9 自动质检

系统至少检查：

- 帧尺寸是否正确。
- 是否存在越界或裁切。
- Alpha 是否符合规则。
- 是否使用非法颜色。
- 角色高度和宽度变化是否超出阈值。
- 脚底锚点是否漂移。
- 身体重心是否异常跳动。
- 相邻帧轮廓变化是否异常。
- 是否存在重复帧。
- 循环首尾差异是否过大。
- 武器或装备尺寸是否异常。
- 角色身份是否疑似变化。

身份、肢体和遮挡检测可以先由视觉模型辅助判断，后续再引入专门的姿势或分割模型。

### 5.10 审核与返工

系统提供：

- 动画实时预览。
- Contact Sheet。
- 原图与规格化结果对比。
- 锚点、包围盒和网格叠加显示。
- 问题帧标记。
- 单帧重新生成。
- 局部区域重新生成。
- 替换外部编辑后的图片。
- 简单像素级修正。
- 版本回退。

### 5.11 组合与导出

首期输出：

- 单帧 PNG。
- Sprite Sheet PNG。
- GIF 或 WebP 动画预览。
- 帧元数据 JSON。
- 质检报告 JSON。
- Godot `SpriteFrames` 或配套导入数据。

后续输出：

- TexturePacker 兼容数据。
- Aseprite 文件或导入脚本。
- Unity Animation Clip 配置。
- 武器、角色和特效分层图集。

## 6. 功能需求

### 6.1 项目管理

- 新建、打开、复制和归档项目。
- 项目自动保存。
- 项目目录可直接备份和迁移。
- 记录项目格式版本，支持后续迁移。
- 项目损坏时尽量从中间资产恢复。

### 6.2 角色描述

- 支持自然语言描述。
- 将自然语言转换为结构化角色设定。
- 用户可以手动修改结构化字段。
- 支持保存负面约束和禁止变化项。
- 支持上传现有角色参考图。

### 6.3 生成任务

- 所有生成任务进入任务队列。
- 支持取消、重试和局部重跑。
- 显示任务状态、耗时和错误。
- 保存模型、参数、提示词和参考输入。
- Provider 失败不能破坏已完成资产。

### 6.4 Provider 抽象

统一接口至少包含：

- `generate_character_reference`
- `generate_variant_reference`
- `generate_keyframe`
- `generate_inbetween`
- `edit_region`
- `inspect_frame`

首期实现 OpenAI Provider，后续支持：

- ComfyUI。
- 本地扩散模型。
- 其他云端图片生成服务。
- 用户手动导入。

### 6.5 规格化流水线

每个处理步骤必须：

- 输入和输出明确。
- 保存中间产物。
- 可以单独重新执行。
- 参数可以写入项目配置。
- 结果具有版本号和来源信息。

### 6.6 动画工作台

- 时间轴。
- 帧列表。
- 播放、暂停、循环和 FPS 调节。
- 洋葱皮显示。
- 网格和锚点显示。
- 帧顺序调整。
- 单帧启用、禁用和替换。
- 原始帧、规格化帧和最终帧切换。

## 7. 产品架构

建议采用本地优先、前后端分离、Provider 可插拔的架构。

```text
FrameGrid Desktop
  |
  +-- UI Layer
  |     项目向导 / 角色编辑 / 动作时间轴 / 质检 / 导出
  |
  +-- Application Layer
  |     项目服务 / 动作规划 / 任务队列 / 版本管理
  |
  +-- Provider Layer
  |     OpenAI / ComfyUI / Local Model / Manual Import
  |
  +-- Image Pipeline
  |     Segment / Normalize / Pixel Snap / Palette / Align / Validate
  |
  +-- Storage
  |     Project Files / SQLite Index / Generated Assets / Reports
  |
  +-- Exporters
        PNG / Sprite Sheet / GIF / JSON / Godot
```

### 7.1 为什么采用本地优先

- 游戏资产通常以本地文件为中心。
- 图片处理中间文件较多，不适合全部依赖云端传输。
- 用户可能使用本地 ComfyUI 或其他模型。
- 更容易和 Godot、Aseprite、Git 等本地工具协作。
- 项目可以完整备份，不依赖 FrameGrid 服务持续在线。

### 7.2 Codex 的职责边界

Codex 官方定位是软件开发编码代理，不是图片生成模型。

FrameGrid 可以在两个位置使用 Codex：

1. 开发阶段：辅助开发 FrameGrid、编写处理脚本和测试。
2. 可选工作流入口：通过 Codex SDK、CLI 或 Skill 让代理操作 FrameGrid 项目和工具。

FrameGrid 正式产品内部的角色规划、提示词组织和图片生成，首选 OpenAI Responses API 与图像生成工具，或者其他 Image Provider。产品不能把核心生图能力绑定在 Codex 桌面应用是否运行。

## 8. 推荐技术栈

### 8.1 桌面应用

- Tauri：桌面外壳、系统菜单、文件选择和应用打包。
- React：主要界面框架。
- TypeScript：前端类型系统。
- Vite：前端构建工具。
- Zustand：轻量界面状态管理。
- TanStack Query：后端任务和项目数据同步。
- Konva.js：画布、网格、锚点、选择框和简单像素编辑。

选择 Tauri 而不是 Electron 的主要原因是安装包和运行时占用更小。MVP 也可以先以浏览器界面运行，再接入 Tauri 打包。

### 8.2 后端与图像流水线

- Python 3.12 或更高稳定版本。
- FastAPI：本地 API 和任务状态接口。
- Pydantic：配置、项目数据和 Provider 输入输出验证。
- Pillow：PNG、Alpha、缩放、调色板和 Sprite Sheet 基础处理。
- NumPy：像素矩阵计算。
- OpenCV：分割、轮廓、形态学操作和图像指标。
- scikit-image：结构相似度、颜色和轮廓分析的补充工具。
- imageio：GIF/WebP 动画预览输出。

核心图像流水线使用 Python，是因为图像处理和机器学习生态最完整，也便于后续接入本地模型。

### 8.3 AI 与生成服务

- OpenAI 官方 Python SDK。
- OpenAI Responses API：动作规划、结构化输出和工具调用。
- Responses API `image_generation` 工具：首期角色参考图和动作帧生成。
- OpenAI Agents SDK：仅在后续确实需要多代理编排、Guardrail 和 Trace 时引入。
- Provider Adapter：隔离 OpenAI、ComfyUI 和其他服务的差异。

MVP 不建议一开始引入复杂多代理系统。角色项目状态机和图像流水线应由确定性代码控制，模型只负责适合模型的规划、生成和视觉判断。

### 8.4 数据与存储

- SQLite：应用项目索引、任务队列和运行记录。
- SQLAlchemy：数据库访问。
- Alembic：数据库结构迁移。
- JSON：可移植项目配置和帧元数据。
- PNG：无损图片和中间资产。

项目真实资产保存在普通目录中，SQLite 不存储大图片，只保存索引和任务状态。

建议项目目录：

```text
my-character.framegrid/
  project.json
  character/
    definition.json
    references/
    palettes/
  variants/
  actions/
    run_down/
      action.json
      raw/
      segmented/
      normalized/
      corrected/
      final/
      reports/
  exports/
  history/
```

### 8.5 后台任务

MVP：

- Python 进程内任务队列。
- SQLite 持久化任务状态。
- 单机并发限制。

后续：

- Dramatiq 或 Celery。
- Redis。
- 云端生成节点和本地处理节点分离。

MVP 不应强制用户安装 Redis。

### 8.6 测试与质量

- pytest：Python 单元测试和流水线测试。
- Hypothesis：图像参数和配置的属性测试。
- Vitest：前端单元测试。
- Playwright：关键界面流程测试。
- Ruff：Python lint 和格式化。
- mypy 或 Pyright：Python 类型检查。
- ESLint 和 Prettier：前端检查与格式化。
- GitHub Actions：自动测试和构建。

### 8.7 开发与依赖管理

- uv：Python 依赖和虚拟环境。
- pnpm：前端依赖管理。
- Git：源码和文档版本管理。
- Git LFS：仅在确实需要提交大型测试图片时使用。

## 10. 状态与版本规则

建议帧状态：

```text
planned
generating
generated
normalizing
needs_review
approved
rejected
exported
failed
```

任何生成或处理操作都不直接覆盖上一版本。新结果必须创建新版本，用户确认后再设为当前版本。

生成记录至少包含：

- Provider。
- 模型标识。
- 时间。
- 请求参数。
- 原始提示词。
- Provider 修改后的提示词，如果可获得。
- 输入参考图。
- 输出文件。
- 错误信息。

## 11. MVP 范围

第一版只完成一条最短但完整的生产闭环。

### 11.1 MVP 输入

- 创建一个角色项目。
- 输入角色描述。
- 选择 `32x32` 或 `64x64` 画布。
- 生成并确认四方向角色参考图。
- 创建一个服装变体属于后续增强，不作为首轮必需项。
- 创建一个 `run_down` 动作。
- 动作固定为 6 帧。

### 11.2 MVP 能力

- OpenAI Image Provider。
- 角色母版生成与选择。
- 关键帧生成。
- 中间帧生成或手动导入。
- 背景清理。
- 严格逻辑像素网格。
- 共享调色板。
- 脚底锚点对齐。
- 帧尺寸和轮廓基础检测。
- 动画预览。
- 单帧替换。
- PNG、Sprite Sheet、GIF 和 JSON 导出。

### 11.3 MVP 暂不包含

- 多角色同屏。
- 复杂装备分层。
- 自动骨骼识别。
- 云端团队协作。
- 插件市场。
- Unity 导出。
- 完整像素画编辑器。
- 多代理自动返工。

## 12. MVP 验收标准

- 用户可以创建、关闭并重新打开项目。
- 项目目录可以复制到另一台机器继续使用。
- 可以从角色描述得到一套可确认的方向参考图。
- 可以创建并预览一个 6 帧 `run_down` 动作。
- 所有最终帧尺寸完全一致。
- 所有最终帧使用统一逻辑像素网格。
- 脚底锚点没有非预期漂移。
- 角色高度和躯干宽度变化可以被量化并标记。
- 所有最终帧符合共享调色板规则。
- 用户可以只替换一个失败帧。
- Sprite Sheet 和元数据可以被 Godot 导入或按规则读取。
- 生成失败、断网或应用重启不会丢失已完成结果。

## 14. 主要风险

### 14.1 角色身份漂移

规格化无法解决身份漂移。需要通过母版、变体参考图、多轮编辑、视觉质检和人工确认共同控制。

### 14.2 动作缺乏连续性

必须先规划关键姿势，再生成中间帧，并检测首尾循环和相邻帧变化。

### 14.3 自动居中破坏动作位移

对齐策略必须区分：

- 非预期漂移。
- 角色主动位移。
- 武器或特效导致的包围盒变化。

不能简单按每帧包围盒独立居中。

### 14.4 像素规格定义不清

画布尺寸、角色占用尺寸和逻辑像素倍率必须分别存储，不能只记录“32 像素角色”。

### 14.5 Provider 锁定

所有生成与检测调用必须经过 Adapter，项目数据中不能出现只有单一 Provider 才能解释的核心字段。

### 14.6 过早追求全自动

首期应优先保证可审查、可替换和可恢复。自动返工系统建立在足够多的失败样本和明确指标之上。

## 15. 首轮需要验证的产品决策

在正式开发桌面界面前，应通过技术原型回答：

1. 角色母版采用四张独立图片还是一张角色转面表更稳定。
2. 生成时使用高分辨率后规格化，还是要求模型直接模拟低分辨率像素画。
3. 逻辑像素网格应自动检测还是由项目固定。
4. 共享调色板按角色、变体还是动作管理。
5. 脚底锚点自动检测的可靠性是否足够，是否必须允许人工设置。
6. 中间帧由图片编辑生成还是由关键帧插值后再重绘。
7. 身份一致性检测应使用通用视觉模型还是专门特征模型。
8. Godot 首期采用 `.tres` 生成还是 JSON 加导入插件。

## 16. 当前建议

FrameGrid 应先实现“确定性的资产流水线”，再逐步增加 AI 自主程度。

推荐优先级：

1. 定义可长期演进的项目格式。
2. 用真实样本验证规格化质量。
3. 实现单动作端到端闭环。
4. 建立失败分类和可视化审核。
5. 再制作桌面工作台。
6. 最后扩展服装变体、更多动作和更多 Provider。

不建议第一步就制作完整 UI，也不建议把所有步骤交给一个 Agent 自行决定。图像资产生产需要明确状态、可重复参数和版本记录，这些应由普通程序控制。
