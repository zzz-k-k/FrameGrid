# FrameGrid 项目上下文

## 1. 项目起点

FrameGrid 来源于一个 Godot 俯视角动作肉鸽项目的美术资产需求。

原游戏具有以下特点：

- 俯视角、四方向自由移动。
- 世界地块以 `32x32` 网格组织。
- 玩家动画按 `idle/run/attack x up/down/left/right` 组织。
- 当前角色动画帧画布为 `96x80`。
- 后续还需要挖掘、受伤、死亡等动作。

目标是探索一条可复用的 AI 像素角色动画生产流程，而不是只为单个角色手工制作一次资产。

## 2. 项目目标

建立一条可重复执行的流水线：

```text
AI 角色设计与关键姿势生成
  -> 自动切帧
  -> 像素规格化
  -> 动画一致性检测
  -> 少量人工修帧或自动返工
  -> 导出游戏引擎可直接使用的资产
```

第一阶段优先支持：

- 四方向角色。
- `idle`、`run`、`attack`、`dig`。
- `32x32`、`64x64` 或固定自定义帧画布。
- PNG Sprite Sheet。
- Godot `SpriteFrames` 资源导出。

## 3. 核心判断

### 3.1 ChatGPT 图像生成是否可行

可行，但不适合让模型彼此独立地生成每一帧。

更可靠的方式是：

1. 固定角色母版。
2. 建立正面、背面和侧面标准帧。
3. 按方向和动作拆分任务。
4. 使用角色母版、上一帧和目标姿势作为编辑条件。
5. 先生成关键帧，再生成中间帧。
6. 使用脚本完成机械规格化和质量检测。
7. 对失败帧局部重做或人工修正。

### 3.2 像素画并非对误差不敏感

低分辨率意味着每个像素对轮廓、肢体和重心的影响更大。

一到两个像素的变化就可能造成：

- 角色轮廓闪烁。
- 头部或身体忽大忽小。
- 武器长度变化。
- 脚底漂移。
- 动画重心跳动。

因此，像素规格化只能解决部分问题，不能替代帧间语义一致性。

### 3.3 脚本适合解决的问题

- 固定帧画布和图集布局。
- 透明背景与 Alpha 二值化。
- 最近邻缩放。
- 统一逻辑像素网格。
- 全动画共享调色板量化。
- 脚底锚点对齐。
- 角色占用面积、包围盒和重心检测。
- 帧间轮廓变化检测。
- Sprite Sheet 打包。
- Godot `SpriteFrames` 生成。

### 3.4 脚本难以独立解决的问题

- 衣服、脸或装备在不同帧发生变化。
- 武器长度和握持关系变化。
- 肢体拓扑错误。
- 遮挡关系错误。
- 动作缺少蓄力、接触和回收。
- 运动方向和步态逻辑错误。

这些问题需要生成约束、视觉模型检查、重新生成或人工修帧。

## 4. 对 Character Animation Creator Skill 的分析

参考项目：

- <https://github.com/tachikomared/character-animation-creator-skill>
- <https://github.com/tachikomared/TachiSnap>

该项目的优点：

- 先建立标准角色。
- 按动作、方向和尺寸分别生成。
- 每个方向独立返工。
- 自动切分和组装 Sprite Sheet。
- 清理绿幕与透明边缘。
- 检查尺寸、裁切、重复帧和运动量。
- 生成 GIF、接触表和质量报告。

它更接近一套 AI 资产生产脚手架，而不是已经彻底解决像素动画一致性的系统。

### 4.1 默认规格化的局限

其 `pixel_snap.py` 主要执行：

- 绿幕清除。
- Alpha 二值化。
- 调色板量化。
- 可选的全图缩小后最近邻放大。

默认并不会自动将模型输出映射到严格、统一的像素网格。

另外，逐帧按包围盒独立裁剪、缩放和居中可能产生：

- 脚底锚点漂移。
- 角色尺寸变化。
- 武器较长的帧被整体缩小。
- 跳跃或攻击位移被居中操作抵消。

### 4.2 TachiSnap 的作用

TachiSnap 更接近真正的像素网格规格化：

- 检测视觉像素尺寸。
- 估算网格步长。
- 将边界吸附到统一网格。
- 使用感知颜色空间进行量化。

但它仍不能修复角色身份、肢体关系和动作节奏等语义问题。

### 4.3 FrameGrid 的改进方向

FrameGrid 应在参考方案基础上重点补足：

1. 全动作共享的像素网格与网格原点。
2. 全角色共享的调色板。
3. 固定脚底锚点，而非逐帧几何居中。
4. 固定身体比例与装备尺寸范围。
5. 对角色本体、武器和特效分别建立图层或检测规则。
6. 检测帧间身份变化，而不只是像素差异。
7. 使用动作关键点或姿势模板约束生成。
8. 直接导出 Godot 资源。

## 5. 建议架构

```text
framegrid/
  docs/
  prompts/
  references/
  src/
    generation/
    segmentation/
    normalization/
    alignment/
    validation/
    exporters/
  tests/
  examples/
```

建议将流水线拆成以下独立阶段：

1. `generate`：通过图像模型生成角色母版、关键帧或动画条。
2. `segment`：识别并切分各帧，保留原始空间关系。
3. `normalize`：统一透明度、像素网格和调色板。
4. `align`：按脚底、身体中心或指定骨骼点对齐。
5. `validate`：检测尺寸、轮廓、颜色、重心、重复帧和异常变化。
6. `review`：生成 GIF、Contact Sheet 和问题报告。
7. `export`：导出 Sprite Sheet、JSON 和 Godot `.tres`。

每个阶段都应保存中间产物，避免一次失败后重新执行整条流水线。

## 6. 建议 MVP

第一版不要立即覆盖所有动作和方向。

建议只处理：

- 一个固定角色。
- `run_down` 六帧。
- 固定 `64x64` 帧画布。
- 固定脚底锚点。
- 一套共享调色板。
- 输出 PNG、GIF、JSON 检测报告。

MVP 的验收标准：

- 所有帧共享同一个逻辑像素网格。
- 脚底位置不发生非预期漂移。
- 角色高度和躯干宽度变化不超过设定阈值。
- 调色板完全一致。
- 动画循环首尾连续。
- 可以快速定位并替换单个失败帧。

完成单方向跑步后，再逐步扩展到四方向、攻击和 Godot 导出。

## 7. 下一步

1. 确定 MVP 的目标帧尺寸和逻辑像素尺寸。
2. 确定图像生成入口：ChatGPT 手动生成、OpenAI API 或可替换 Provider 接口。
3. 定义中间数据格式，包括帧、方向、动作、锚点和调色板。
4. 创建第一套角色参考图和 `run_down` 测试输入。
5. 实现切帧、网格吸附、调色板统一和脚底对齐。
6. 生成 GIF 与检测报告。
7. 评估失败类型，再决定是否引入姿势估计或视觉模型质检。
